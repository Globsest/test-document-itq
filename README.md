# ТЕстовое задание для ITQ Group

### 1. PostgreSQL

```bash
docker-compose up -d
```

Параметры задаются в `docker-compose.yml` и в `src/main/resources/application.properties`.

### 2. Сервис

Сервис слушает порт 8080. После старта применяются миграции Liquibase.

### 3. Утилита массового создания документов

Утилита читает число N из файла параметров и создаёт N документов через API.

**Параметры** — в `src/main/resources/generator.properties`:
- `count` — количество документов (по умолчанию 100)
- `api.baseUrl` — URL сервиса (по умолчанию http://localhost:8080)
- `author`, `initiator` — подставляются в запросы создания

**Запуск** (сервис должен быть уже запущен):

```bash
./gradlew runGenerator
```

## Проверка прогресса по логам

- **Утилита генерации:** в консоли выводится «Задано к созданию документов: N = …», затем «Прогресс создания: X / N документов» (каждые 100 и в конце), в конце — «Создание завершено за … мс: создано …, ошибок …».
- **SUBMIT-worker:** в логах сервиса — «SUBMIT-worker: найдено DRAFT: …, обрабатываю пачку …», затем «SUBMIT-worker: отправлена пачка … документов за … мс, успешно …, ошибок …, осталось DRAFT: …».
- **APPROVE-worker:** «APPROVE-worker: найдено SUBMITTED: …, обрабатываю пачку …», затем «APPROVE-worker: утверждена пачка … документов за … мс, успешно …, ошибок …, осталось SUBMITTED: …».

Частичные ошибки в пачке не останавливают воркеры; размер пачки задаётся в `application.properties`: `workers.submit.batchSize`, `workers.approve.batchSize`.

## Поиск документов и период дат

В API поиска фильтр по периоду дат трактуется **по дате создания документа** (`created_at`). Параметры `dateFrom` и `dateTo` задают границы интервала по полю создания. См. `EXPLAIN.md` для примера запроса и индексов.

## API 

- `POST /api/documents` — создать документ (DRAFT, номер генерируется).
- `GET /api/documents/{id}` — один документ с историей.
- `GET /api/documents?ids=1,2,3` — пакетное получение по id.
- `GET /api/documents/search` — поиск с фильтрами (статус, автор, период по дате создания), пагинация и сортировка.
- `POST /api/documents/submit` — отправить на согласование (список id, DRAFT → SUBMITTED).
- `POST /api/documents/approve` — утвердить (SUBMITTED → APPROVED, запись в реестр; при ошибке записи в реестр — откат).
- `POST /api/documents/test-concurrent-approval` — тест конкурентного утверждения (threads, attempts, сводка и финальный статус).

Формат ошибок: код + сообщение


## Опционально (масштабирование и вынос реестра)

**Обработка одного запроса с 5000+ id:**  
Разбивать входящий список на чанки (по 500–1000 id), обрабатывать чанки последовательно или через ограниченный пул потоков; при необходимости — асинхронный приём запроса и ответ по результатам (через polling или callback).
